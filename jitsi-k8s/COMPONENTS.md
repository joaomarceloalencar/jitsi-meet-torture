# Jitsi Meet Kubernetes Components

This document provides a quick reference for all components in the Jitsi Meet Kubernetes deployment.

## Directory Structure

```
jitsi-k8s/
├── certs/              # TLS certificates (generated by scripts)
├── k8s/                # Kubernetes manifests
├── scripts/            # Helper scripts
├── .secrets/           # JWT secrets (generated)
└── README.md           # Main documentation
```

## Kubernetes Manifests

### 00-namespace.yaml
- **Purpose**: Creates the `jitsi` namespace
- **Resources**: 1 Namespace

### 01-nginx-ingress-controller.yaml
- **Purpose**: Deploys nginx-ingress-controller for routing
- **Resources**:
  - 1 Namespace (ingress-nginx)
  - 1 ServiceAccount
  - 1 ClusterRole + ClusterRoleBinding
  - 1 Role + RoleBinding
  - 3 ConfigMaps
  - 1 Service (LoadBalancer)
  - 1 Deployment
  - 1 IngressClass

### 02-configmaps.yaml
- **Purpose**: Configuration for all Jitsi components
- **Resources**: 5 ConfigMaps
  - `jitsi-common-config`: Shared configuration (XMPP, JWT, etc.)
  - `jitsi-web-config`: Web interface settings
  - `jitsi-prosody-config`: XMPP server settings
  - `jitsi-jicofo-config`: Conference focus settings
  - `jitsi-jvb-config`: Videobridge settings

### 03-secrets.yaml.template
- **Purpose**: Template for secrets (populated by setup.sh)
- **Resources**: 3 Secrets
  - `jitsi-tls`: TLS certificate and key
  - `jitsi-jwt`: JWT application secret and key
  - `jitsi-passwords`: Component passwords

### 04-deployments.yaml
- **Purpose**: Deploys all Jitsi components
- **Resources**: 4 Deployments
  - `jitsi-web`: Web interface (1 replica)
  - `prosody`: XMPP server (1 replica)
  - `jicofo`: Conference focus (1 replica)
  - `jvb`: Videobridge (2 replicas, auto-scalable)

### 05-services.yaml
- **Purpose**: Exposes Jitsi components
- **Resources**: 5 Services
  - `jitsi-web`: HTTP/HTTPS (ClusterIP)
  - `prosody`: XMPP ports (ClusterIP)
  - `jicofo`: HTTP API (ClusterIP)
  - `jvb`: Colibri API (ClusterIP)
  - `jvb-udp`: RTP/UDP (NodePort 30000)

### 06-hpa.yaml
- **Purpose**: Auto-scales JVB based on CPU/memory
- **Resources**: 1 HorizontalPodAutoscaler
  - Min replicas: 2
  - Max replicas: 10
  - Target CPU: 70%
  - Target Memory: 80%

### 07-ingress.yaml
- **Purpose**: Routes HTTPS traffic to jitsi-web
- **Resources**: 1 Ingress
  - Host: jitsi.macbookpro
  - TLS: Enabled with custom certificate
  - Backend: jitsi-web service

## Scripts

### generate-ca.sh
- **Purpose**: Generate Certificate Authority
- **Outputs**:
  - `certs/ca-cert.pem`: CA certificate
  - `certs/ca-key.pem`: CA private key

### generate-certs.sh
- **Purpose**: Generate TLS certificates for jitsi.macbookpro
- **Requirements**: CA must exist
- **Outputs**:
  - `certs/tls.crt`: Server certificate
  - `certs/tls.key`: Server private key

### install-ca-macos.sh
- **Purpose**: Install CA certificate in macOS System Keychain
- **Requirements**: sudo access
- **Actions**: Adds CA to trusted certificates

### setup.sh
- **Purpose**: Automated deployment of entire stack
- **Actions**:
  1. Checks prerequisites (kubectl, openssl)
  2. Generates certificates if needed
  3. Generates random passwords and JWT secrets
  4. Creates secrets manifest from template
  5. Prompts to add /etc/hosts entry
  6. Deploys all Kubernetes resources in order
  7. Waits for pods to be ready

### generate-jwt-token.sh
- **Purpose**: Generate JWT tokens for authentication
- **Usage**: `./generate-jwt-token.sh <username> <moderator|user>`
- **Requirements**: Python 3, PyJWT
- **Output**: JWT token valid for 24 hours

### cleanup.sh
- **Purpose**: Remove all Kubernetes resources
- **Actions**:
  1. Deletes Ingress
  2. Deletes HPA
  3. Deletes Deployments
  4. Deletes Services
  5. Deletes ConfigMaps
  6. Deletes Secrets
  7. Deletes Namespace
  8. Optionally deletes nginx-ingress-controller
- **Note**: Preserves certificates and secrets on disk

## Security

### JWT Authentication
- **Algorithm**: HS256
- **Issuer**: jitsi-meet
- **Audience**: jitsi-meet
- **Token Lifetime**: 24 hours
- **Secret Storage**: Kubernetes Secret

### TLS/HTTPS
- **Certificate**: Self-signed (for local)
- **CA**: Local CA installed in System Keychain
- **Protocol**: TLS 1.2, TLS 1.3
- **Termination**: nginx-ingress-controller

### Passwords
All passwords are randomly generated 64-character hex strings:
- JICOFO_AUTH_PASSWORD
- JVB_AUTH_PASSWORD
- JIGASI_XMPP_PASSWORD
- JIBRI_RECORDER_PASSWORD
- JIBRI_XMPP_PASSWORD

## Scaling

### Auto-scaling (HPA)
- **Component**: jvb
- **Min Replicas**: 2
- **Max Replicas**: 10
- **Metrics**: CPU (70%), Memory (80%)
- **Scale-up**: Fast (100% or 2 pods per 30s)
- **Scale-down**: Conservative (50% or 1 pod per 60s)
- **Stabilization**: 5 minutes before scale-down

### Manual Scaling
```bash
# Scale JVB
kubectl scale deployment jvb -n jitsi --replicas=<number>

# Scale other components (not recommended)
kubectl scale deployment jitsi-web -n jitsi --replicas=<number>
kubectl scale deployment prosody -n jitsi --replicas=<number>
kubectl scale deployment jicofo -n jitsi --replicas=<number>
```

**Note**: Prosody and Jicofo are not designed for horizontal scaling and should remain at 1 replica.

## Troubleshooting Quick Checks

```bash
# 1. Check if all pods are running
kubectl get pods -n jitsi

# 2. Check pod logs for errors
kubectl logs -n jitsi <pod-name>

# 3. Check ingress
kubectl describe ingress jitsi-ingress -n jitsi

# 4. Check /etc/hosts
grep jitsi.macbookpro /etc/hosts

# 5. Check certificate
security find-certificate -c "Jitsi Local CA" -a

# 6. Test connectivity
curl -k https://jitsi.macbookpro

# 7. Check HPA
kubectl describe hpa jvb-hpa -n jitsi

# 8. Check events
kubectl get events -n jitsi --sort-by='.lastTimestamp'
```
